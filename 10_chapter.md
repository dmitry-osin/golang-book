### Глава 10. Обработка ошибок

#### 1. Тип `error` и Идиоматичная обработка
Встроенный тип `error` в Go — это просто интерфейс с одним единственным методом `Error()`, который возвращает текст ошибки.

```go
type error interface {
    Error() string
}
```

*Пример из жизни:* Ты прыгаешь с парашютом. В парадигме `try/catch` ты просто выпрыгиваешь из самолета и надеешься, что если основной парашют не раскроется, сработает автоматический блок `catch` и выпустит запасной. В парадигме Go ты **явно** проверяешь каждый карабин перед прыжком: `if err != nil`. Да, кода становится больше, но сюрпризов — меньше.

```go
package main

import (
    "errors"
    "fmt"
)

// Пользовательская ошибка
var ErrDivideByZero = errors.New("ошибка: деление на ноль")

func divide(a, b int) (int, error) {
    if b == 0 {
        return 0, ErrDivideByZero // Возвращаем конкретную ошибку
    }
    return a / b, nil // nil означает "ошибки нет"
}

func main() {
    result, err := divide(10, 0)
    // Идиоматичная (классическая) проверка ошибки в Go
    if err != nil {
        fmt.Println("Что-то пошло не так:", err)
        return // Прерываем выполнение
    }
    fmt.Println("Результат:", result)
}
```

#### 2. Оборачивание ошибок (Error Wrapping)
В сложных системах ошибка может пройти через 5 слоев архитектуры (БД -> Репозиторий -> Сервис -> HTTP-контроллер). Если база вернет "not found", а контроллер просто отдаст это клиенту, мы никогда не поймем, **где именно** произошел сбой.

Ошибки нужно "оборачивать", добавляя контекст. 
Для этого используется глагол `%w` (wrap) в функции `fmt.Errorf`.



```go
import (
    "errors"
    "fmt"
    "os"
)

func readFile(filename string) error {
    _, err := os.Open(filename)
    if err != nil {
        // Оборачиваем оригинальную ошибку (err) в нашу кастомную.
        // Теперь это "ошибка-матрешка".
        return fmt.Errorf("ошибка чтения конфига: %w", err)
    }
    return nil
}

func main() {
    err := readFile("config.json")
    if err != nil {
        fmt.Println(err) // Выведет: ошибка чтения конфига: open config.json: no such file or directory
        
        // Как проверить, какая оригинальная ошибка лежит внутри матрешки?
        // Функция errors.Is распаковывает ошибку и проверяет совпадения.
        if errors.Is(err, os.ErrNotExist) {
            fmt.Println("Точная причина: файла не существует!")
        }
    }
}
```
*(Также существует `errors.As`, которая позволяет извлечь из "матрешки" конкретную структуру ошибки, чтобы прочитать её внутренние поля).*

#### 3. Оператор `defer` (Отложенное выполнение)
При работе с файлами, сетью или базой данных мы обязаны освобождать ресурсы (закрывать соединения), даже если произошла ошибка. В других языках для этого используют блок `finally`. В Go придумали гениально простой оператор `defer`.

`defer` откладывает выполнение функции до момента выхода из текущей функции (неважно, произошел этот выход через `return` или из-за ошибки).

```go
func processFile(filename string) error {
    file, err := os.Open(filename)
    if err != nil {
        return err
    }
    
    // defer регистрируется СРАЗУ ПОСЛЕ успешного открытия.
    // Вызов file.Close() сработает ровно перед тем, как функция processFile завершится.
    defer file.Close() 

    // Дальше мы пишем логику чтения файла, вообще не думая о том, 
    // что его нужно закрыть. Если тут произойдет куча return из-за ошибок,
    // defer все равно сработает.
    fmt.Println("Читаем файл...")
    
    return nil
}
```
**Важно:** Несколько вызовов `defer` выстраиваются в стек (LIFO — "последним пришел, первым ушел"). Если ты напишешь `defer A`, затем `defer B`, то при выходе сначала выполнится `B`, а потом `A`.

#### 4. Паника (`panic`) и Восстановление (`recover`)
Иногда происходит что-то настолько ужасное, что программа физически не может продолжать работу. Например:
* Обращение к памяти по нулевому указателю (Nil pointer dereference).
* Выход за пределы массива (Index out of range).

В этом случае Go вызывает **панику (`panic`)**. Программа бросает текущую работу, начинает раскручивать стек (выполняя все `defer`) и жестко падает. Ты можешь вызвать панику и вручную: `panic("Невозможно подключиться к базе!")`.

Но в web-серверах нельзя допускать, чтобы из-за одного кривого запроса упал весь сервер, обслуживающий тысячи других пользователей. Для этого используется `recover`.

```go
func safeWorker() {
    // defer с функцией-спасателем
    defer func() {
        // recover() перехватывает панику. Если паники не было, вернет nil.
        if r := recover(); r != nil {
            fmt.Println("Спасены от паники! Причина:", r)
            // Здесь мы можем залогировать ошибку и позволить программе жить дальше
        }
    }()

    fmt.Println("Начинаем опасную работу...")
    
    // Искусственно вызываем панику. Код ниже этой строки не выполнится.
    panic("Что-то взорвалось!") 
    
    fmt.Println("Этот текст мы никогда не увидим.")
}

func main() {
    safeWorker()
    fmt.Println("Программа штатно завершила работу.")
}
```
*Правило эксперта:* Никогда не используй `panic` для обычного управления потоком программы (вместо `if err != nil`). Оставь панику только для реально критических багов (например, невозможность прочитать стартовый конфиг при запуске приложения).

#### 5. Контекст (`context.Context`)
Это важнейший паттерн для Highload.
*Пример из жизни:* Представь, что менеджер (клиент) поручил команде (серверу) сделать отчет. Команда запустила 5 параллельных задач. Через секунду менеджер звонит и говорит: "Отмена, отчет больше не нужен". Без механизма отмены команда продолжит впустую жечь процессор и грузить базу данных. 

`context` позволяет передавать сигнал об отмене (или таймауте) сквозь всю цепочку вызовов функций и горутин.

```go
import (
    "context"
    "fmt"
    "time"
)

func doWork(ctx context.Context) {
    select {
    case <-time.After(5 * time.Second):
        fmt.Println("Работа успешно завершена")
    case <-ctx.Done(): // Этот канал закроется, если контекст отменят
        fmt.Println("Работа прервана:", ctx.Err())
    }
}

func main() {
    // Создаем контекст с таймаутом в 2 секунды
    ctx, cancel := context.WithTimeout(context.Background(), 2*time.Second)
    defer cancel() // Обязательно освобождаем ресурсы контекста

    go doWork(ctx)

    // Ждем 3 секунды. Контекст "сгорит" через 2 секунды и пошлет сигнал отмены в doWork.
    time.Sleep(3 * time.Second) 
}
```
