### Глава 15. Промышленное тестирование (Инженерная культура)

#### 1. Идиоматичные Unit-тесты и Table-driven подход
В Go принято писать тесты в файлах с суффиксом `_test.go`, которые лежат в той же папке, что и основной код. 

Вместо того чтобы писать 10 отдельных функций для проверки разных сценариев (как в Java или Python), в Go используется **Table-driven** подход (тестирование на основе таблиц). 

*Пример из жизни:* Представь паспортный контроль в аэропорту. У пограничника есть один «алгоритм» проверки. Он не меняет его для каждого пассажира. Вместо этого перед ним проходит «таблица» (очередь) людей с разными данными. Его задача — применить один и тот же тест к каждому ряду данных.



```go
package mathutils

import "testing"

// Функция, которую мы тестируем
func Add(a, b int) int {
    return a + b
}

// Тестовая функция всегда начинается с Test и принимает *testing.T
func TestAdd(t *testing.T) {
    // 1. Описываем "таблицу" тест-кейсов
    tests := []struct {
        name     string // Описание случая
        a, b     int    // Входные данные
        expected int    // Что мы ожидаем получить
    }{
        {"Положительные", 2, 2, 4},
        {"Отрицательные", -1, -1, -2},
        {"С нулем", 5, 0, 5},
    }

    // 2. Итерируемся по таблице
    for _, tt := range tests {
        // t.Run создает "подтест". Если один упадет, остальные продолжат работу.
        t.Run(tt.name, func(t *testing.T) {
            result := Add(tt.a, tt.b)
            if result != tt.expected {
                // Если результат не совпал, помечаем тест как проваленный
                t.Errorf("Add(%d, %d) = %d; want %d", tt.a, tt.b, result, tt.expected)
            }
        })
    }
}
```

#### 2. Бенчмарки (`go test -bench`)
Эксперт Go всегда знает, сколько наносекунд занимает его код. Бенчмарки позволяют замерить производительность и количество выделяемой памяти.

*Пример из жизни:* Бенчмарк — это секундомер у тренера. Спортсмен (твой код) должен пробежать дистанцию не один раз (это может быть случайностью), а тысячи раз, чтобы тренер вычислил среднее время.

```go
// Бенчмарк начинается с Benchmark и принимает *testing.B
func BenchmarkAdd(b *testing.B) {
    // b.N — это магическое число, которое Go подбирает сам.
    // Он будет запускать цикл столько раз, сколько нужно, 
    // чтобы получить статистически значимый результат.
    for i := 0; i < b.N; i++ {
        Add(10, 20)
    }
}
```
Запуск: `go test -bench=. -benchmem`
Флаг `-benchmem` покажет, сколько байт памяти выделяет код за одну операцию и сколько раз он обращается к «куче» (allocs/op). Для Senior-разработчика идеал — это 0 аллокаций в критических узлах системы.

#### 3. Fuzzing-тестирование (Фаззинг)
Это «тяжелая артиллерия», появившаяся в Go 1.18. Ты не придумываешь данные сам, а заставляешь Go генерировать случайный мусор, чтобы попытаться сломать твою функцию.

*Пример из жизни:* Фаззинг — это как дать телефон маленькому ребенку. Он не будет нажимать кнопки «правильно». Он будет нажимать всё подряд, трясти его и облизывать. Если телефон после этого не завис и не сгорел — значит, он надежный.



```go
func FuzzReverse(f *testing.F) {
    // Начальные данные (seed), от которых Go начнет мутации
    f.Add("ABC") 

    f.Fuzz(func(t *testing.T, orig string) {
        // Go будет подставлять сюда сумасшедшие строки: 
        // пустые, огромные, со странными символами Unicode.
        rev := Reverse(orig)
        doubleRev := Reverse(rev)
        
        // Главное правило: если перевернуть строку дважды, 
        // мы должны получить оригинал.
        if orig != doubleRev {
            t.Errorf("Ошибка для строки %q", orig)
        }
    })
}
```
Фаззинг часто находит ошибки переполнения буфера или паники на странных символах, о которых разработчик никогда бы не подумал.

#### 4. Mock-тестирование и изоляция
Когда твоя функция должна отправить письмо или списать деньги с реальной карты, ты не можешь вызвать её в тесте. Тебе нужно «подменить» реальный объект фальшивкой (**Mock**).

В Go это делается через **Интерфейсы**. 

```go
// 1. Описываем интерфейс отправителя
type EmailSender interface {
    Send(to, msg string) error
}

// 2. В коде принимаем ИНТЕРФЕЙС
func NotifyUser(s EmailSender) {
    s.Send("admin@test.com", "Hello!")
}

// 3. В тесте создаем "пустышку"
type MockSender struct{}
func (m MockSender) Send(to, msg string) error {
    // Ничего не отправляем по сети, просто возвращаем nil
    return nil
}

func TestNotify(t *testing.T) {
    mock := MockSender{}
    NotifyUser(mock) // Тестируем логику без интернета
}
```
В реальных проектах для генерации таких «моков» используют утилиты вроде `mockgen` (Uber) или `testify/mock`, чтобы не писать их руками.
