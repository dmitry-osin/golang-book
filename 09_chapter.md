### Глава 9. Параллельное программирование. Горутины

Прежде всего, важно различать два понятия:
* **Конкурентность (Concurrency):** Умение программы иметь дело со множеством задач одновременно (например, один повар готовит салат, пока мясо жарится само).
* **Параллелизм (Parallelism):** Физическое выполнение нескольких задач в одну и ту же секунду (два повара на кухне делают два разных блюда). 

Go дает вам инструменты и для того, и для другого.

#### 1. Горутины (Goroutines)
**Горутина** — это легковесный поток управления, управляемый средой выполнения Go (Runtime), а не операционной системой.

*Пример из жизни:* * **Потоки ОС (Threads)** в Java или C++ — это как нанять отдельную кейтеринговую компанию для каждой задачи. Это дорого, требует много документов и места (каждый поток занимает около 1-8 МБ памяти). 
* **Горутины** — это просто дополнительные повара на вашей кухне. Они очень легкие (занимают всего 2 КБ в начале), и вы можете запустить их миллионы на одном обычном ноутбуке.

Чтобы запустить функцию в отдельной горутине, достаточно просто написать ключевое слово `go` перед её вызовом.

```go
package main

import (
    "fmt"
    "time"
)

func sayHello() {
    for i := 0; i < 5; i++ {
        fmt.Println("Привет из горутины!")
        time.Sleep(100 * time.Millisecond)
    }
}

func main() {
    // Запускаем функцию асинхронно. Программа не ждет её завершения
    // и сразу переходит к следующей строке.
    go sayHello()

    fmt.Println("Главный поток продолжает работу...")
    
    // Если мы не "притормозим" main, программа завершится раньше, 
    // чем горутина успеет что-то напечатать.
    time.Sleep(600 * time.Millisecond)
    fmt.Println("Конец программы.")
}
```

#### 2. Синхронизация горутин и `WaitGroup`
Использовать `time.Sleep`, чтобы дождаться горутину — это плохой тон. Мы никогда не знаем точно, сколько времени займет задача. Для этого есть пакет `sync` и инструмент `WaitGroup`.

*Пример из жизни:* Это как учитель, который перед отправлением автобуса считает детей по головам. 
1. `Add(1)` — ребенок вышел из школы (задача началась).
2. `Done()` — ребенок зашел в автобус (задача завершена).
3. `Wait()` — автобус стоит и ждет, пока счетчик не станет равным нулю.

```go
import (
    "fmt"
    "sync"
)

func worker(id int, wg *sync.WaitGroup) {
    // defer гарантирует, что Done() вызовется в любом случае в конце функции
    defer wg.Done() 

    fmt.Printf("Воркер %d приступил к работе\n", id)
}

func main() {
    var wg sync.WaitGroup

    for i := 1; i <= 3; i++ {
        wg.Add(1) // Говорим: "Мы ждем еще одну задачу"
        go worker(i, &wg) // Важно передавать wg по указателю!
    }

    wg.Wait() // Ждем, пока все 3 воркера вызовут Done()
    fmt.Println("Все задачи выполнены!")
}
```

#### 3. Каналы (Channels) — «Общайтесь, делясь памятью»
Главный лозунг Go: **«Не общайтесь через общую память; делитесь памятью через общение»**. Вместо того чтобы горутины лезли в одну и ту же переменную (что приводит к хаосу), они передают данные друг другу через «трубы» — **Каналы**.



```go
func main() {
    // Создаем канал для передачи целых чисел
    ch := make(chan int)

    go func() {
        fmt.Println("Горутина: выполняю расчеты...")
        time.Sleep(time.Second)
        ch <- 42 // Отправляем данные в канал (стрелка в канал)
    }()

    // Main будет заблокирован на этой строке, пока данные не придут в канал
    val := <-ch // Получаем данные из канала (стрелка из канала)
    fmt.Println("Получено из канала:", val)
}
```

* **Небуферизированные каналы:** (как в примере выше) — это как передача эстафетной палочки в руки. Отправитель ждет, пока получатель заберет данные, и наоборот. Они синхронизируют горутины.
* **Буферизированные каналы:** `make(chan int, 3)`. Это как почтовый ящик. Ты можешь положить туда 3 письма и уйти по своим делам, не дожидаясь почтальона. Но когда ящик заполнится, тебе придется стоять и ждать, пока его не освободят.

#### 4. Конструкция `select`
Это «коммутатор» для каналов. Он позволяет горутине ждать сразу несколько операций с каналами. 

*Пример из жизни:* Это повар, который смотрит на три кастрюли одновременно. Какая первая закипит, той он и займется.

```go
select {
case msg1 := <-chan1:
    fmt.Println("Пришло из первого источника:", msg1)
case chan2 <- "Ответ":
    fmt.Println("Удалось отправить ответ во второй источник")
case <-time.After(time.Second):
    fmt.Println("Таймаут! Никто не ответил за 1 секунду")
}
```

#### 5. Гонка данных (Data Race) и Мьютексы
Когда две горутины одновременно пытаются изменить одну и ту же переменную (например, баланс счета), возникает **Race Condition**. Результат будет непредсказуемым.

*Пример из жизни:* Два человека одновременно пытаются зайти в узкую дверь туалета. Решение — **Мьютекс (Mutex)**. Это замок на двери. Кто первый схватил замок (`Lock`), тот заходит. Второй стоит и ждет у двери, пока первый не выйдет и не откроет замок (`Unlock`).



```go
type Counter struct {
    mu    sync.Mutex
    value int
}

func (c *Counter) Increment() {
    c.mu.Lock()         // Закрываем дверь на замок
    defer c.mu.Unlock() // Гарантируем открытие двери в конце
    c.value++
}
```

**Экспертный совет:** Всегда проверяйте свой код на наличие гонок командой `go test -race`. Она покажет вам, где вы забыли поставить мьютекс.

#### 6. Оптимизации: `sync.Pool` и `sync/atomic`
* **`sync/atomic`:** Если вам нужно просто прибавить единицу к числу, мьютекс может быть слишком тяжелым (медленным). Атомарные операции делают это на уровне инструкций процессора — это максимально быстро.
* **`sync.Pool`:** Если вы создаете миллионы временных объектов (например, буферов для обработки текста), Garbage Collector сойдет с ума их чистить. `sync.Pool` позволяет «сохранять» использованные объекты и выдавать их заново, не выделяя новую память. Это путь к Zero-Allocation коду.
