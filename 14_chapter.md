### Глава 14. Обобщенное программирование (Generics)

#### 1. Параметры типов (Type Parameters)
Суть дженериков в том, что мы передаем в функцию или структуру не только данные (аргументы), но и **типы** этих данных.

*Пример из жизни:* У тебя есть завод по производству коробок. Без дженериков тебе нужен отдельный чертеж станка для коробок под обувь, отдельный для коробок под торты и отдельный для коробок под телевизоры. Дженерики — это умный 3D-принтер. Ты пишешь **один** универсальный чертеж (шаблон), а в момент запуска просто говоришь принтеру: "Сделай коробку для `Торта`".

Синтаксис параметров типа заключается в квадратные скобки `[...]` перед обычными круглыми скобками аргументов.

```go
package main

import "fmt"

// Функция принимает параметр типа [T any].
// Слово "any" означает, что T может быть абсолютно любым типом.
func PrintSlice[T any](items []T) {
    for _, item := range items {
        fmt.Println(item)
    }
}

func main() {
    // 1. Явное указание типа (вызываем функцию и передаем ей тип int)
    nums := []int{1, 2, 3}
    PrintSlice[int](nums)

    // 2. Вывод типов (Type Inference)
    // Go достаточно умен, чтобы сам понять тип из передаваемых аргументов.
    // Квадратные скобки можно опустить!
    words := []string{"hello", "world"}
    PrintSlice(words) 
}
```

#### 2. Ограничения типов (Constraints)
Если мы используем `any`, компилятор не разрешит нам делать с переменной ничего, кроме как передавать её дальше. Мы не можем сложить два числа типа `any`, потому что под `any` может скрываться структура, а структуры складывать нельзя.

Нам нужно ограничить наш 3D-принтер: "Делай коробки для чего угодно, **но только** если это прямоугольное". Для этого используются ограничения (Constraints) — по сути, это интерфейсы нового поколения.



В Go есть встроенное ограничение **`comparable`**. Оно означает "любой тип, который можно сравнить с помощью `==` и `!=`". (Это все числа, строки, булевы значения, но НЕ слайсы и не функции).

```go
// T должно быть comparable, чтобы мы могли сравнивать target == item
func FindIndex[T comparable](slice []T, target T) int {
    for i, item := range slice {
        if item == target {
            return i
        }
    }
    return -1 // Не найдено
}
```

**Кастомные ограничения:**
Если нам нужна математика (сложение, вычитание), мы можем создать свое ограничение, перечислив разрешенные типы через вертикальную черту `|` (знак объединения).

```go
// Описываем интерфейс-ограничение
type Number interface {
    int | int64 | float64
}

// Теперь T может быть только одним из типов, перечисленных в Number
func Sum[T Number](a, b T) T {
    return a + b
}

func main() {
    fmt.Println(Sum(5, 10))         // Работает для int (15)
    fmt.Println(Sum(3.14, 2.0))     // Работает для float64 (5.14)
    // Sum("a", "b") // ОШИБКА КОМПИЛЯЦИИ: string не является Number
}
```

**Магия символа тильда `~`:**
Вспомни Главу 5, где мы создавали кастомные типы (`type Amount int64`). Наш интерфейс `Number` содержит базовый `int64`, но он **не пропустит** пользовательский тип `Amount`.
Чтобы разрешить не только сам тип, но и все производные от него типы, перед именем типа ставится тильда:
`type Number interface { ~int | ~int64 | ~float64 }`. 
Теперь `Sum` сможет складывать и балансы банковских счетов!

*(Примечание: в библиотеке `golang.org/x/exp/constraints` уже написаны все нужные ограничения для чисел и сортировок, их не нужно писать руками).*

#### 3. Универсальные структуры данных
Дженерики можно применять не только к функциям, но и к структурам. Это открывает путь к созданию классических универсальных коллекций: списков, стеков, деревьев.

Давай напишем универсальный Стек (Stack - работает по принципу LIFO "последним пришел - первым ушел"), который можно будет использовать для любых данных.

```go
// Структура имеет параметр типа T
type Stack[T any] struct {
    elements []T
}

// В методах мы обязаны указывать [T] у ресивера (получателя)
func (s *Stack[T]) Push(value T) {
    s.elements = append(s.elements, value)
}

func (s *Stack[T]) Pop() (T, bool) {
    if len(s.elements) == 0 {
        var zero T // Создаем нулевое значение (Zero Value) для типа T
        return zero, false
    }
    
    // Получаем последний элемент
    index := len(s.elements) - 1
    element := s.elements[index]
    
    // Удаляем его из слайса
    s.elements = s.elements[:index]
    
    return element, true
}

func main() {
    // Создаем стек для чисел
    intStack := Stack[int]{}
    intStack.Push(10)
    intStack.Push(20)
    val, _ := intStack.Pop()
    fmt.Println("Извлекли:", val) // 20

    // Создаем стек для строк (тот же самый код структуры!)
    stringStack := Stack[string]{}
    stringStack.Push("Hello")
    stringStack.Push("World")
}
```

#### 4. Как это работает под капотом (Архитектура эксперта)
Многие боятся дженериков, потому что в Java они вызывают стирание типов (Type Erasure) и проблемы в рантайме, а в C++ (шаблоны) они раздувают размер скомпилированного файла (Monomorphization), так как компилятор копирует код функции для каждого типа отдельно.

Инженеры Go пошли гибридным путем, который называется **GCshape stenciling**. 
* Когда компилятор видит `PrintSlice[int]` и `PrintSlice[float64]`, он генерирует **две разные функции** в бинарном файле, потому что числа лежат в памяти по-разному.
* Но если ты вызываешь `PrintSlice[*User]` и `PrintSlice[*Order]` (с указателями), компилятор генерирует **только одну функцию**, потому что под капотом процессора любой указатель — это просто 8-байтное число (на 64-битной системе). Их "форма в памяти" (GCshape) идентична.

Это дает идеальный баланс: дженерики в Go работают так же быстро, как жестко типизированный код (без оверхеда на рефлексию), но при этом бинарный файл не раздувается до гигантских размеров из-за указателей.

**Золотое правило:** "Интерфейсы — когда объекты имеют общее **поведение** (разные способы сварить кофе). Дженерики — когда объекты имеют общую **структуру данных**, не зависящую от их содержимого (коробка, в которую можно положить всё что угодно)".
