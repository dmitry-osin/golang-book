### Глава 16. Производительность под капотом (Under the Hood)

#### 1. Сборщик мусора (Garbage Collector) и алгоритм Tricolor
В языках без сборщика мусора (C, C++, Rust) ты сам выделяешь память и сам обязан её освободить. Если забыл — происходит утечка памяти (Memory Leak), и сервер падает. В Go памятью в «куче» (Heap) управляет **Garbage Collector (GC)**.

Главный враг любой Highload-системы — это фаза **STW (Stop The World)**. Это момент, когда сборщик мусора полностью ставит на паузу все горутины, чтобы безопасно очистить память. В ранних версиях Java такие паузы могли длиться секундами. В Go инженеры сделали ставку на минимальные задержки (Low Latency) — паузы STW в Go редко превышают 1 миллисекунду!

Для этого используется конкурентный алгоритм **Tricolor Mark-and-Sweep** (Трехцветная маркировка), который работает параллельно с твоим кодом.

*Пример из жизни:* Представь, что GC — это уборщик в офисе. Ему нужно выкинуть все ненужные документы. 
Плохой GC кричит: «Всем замереть!» (Stop The World), обходит все столы, собирает мусор и только потом разрешает работать дальше.
Хороший GC (как в Go) маркирует документы прямо во время работы людей:
* **Белые (White):** Кандидаты на удаление (мусор). Изначально все объекты — белые.
* **Серые (Grey):** Уборщик нашел документ (он нужен), но еще не проверил скрепленные с ним приложения.
* **Черные (Black):** Документ и все его приложения проверены. Он 100% нужен для работы, его выкидывать нельзя.



В конце цикла всё, что осталось Белым, безжалостно удаляется из памяти.

*Инженерный хак:* Ты можешь управлять агрессивностью GC с помощью переменной окружения `GOGC`. По умолчанию `GOGC=100` (GC запускается, когда размер кучи удваивается). Если поставить `GOGC=200`, GC будет запускаться реже, потребляя меньше процессора, но программа будет "съедать" больше оперативной памяти.

#### 2. Escape Analysis (Анализ утечек)
Мы обсуждали стек и кучу в Главе 4. Стек — это быстрая временная память функции, куча — медленная глобальная память, которую как раз и должен чистить Garbage Collector.

Откуда компилятор Go знает, куда положить твою переменную? Он использует **Escape Analysis**.

*Пример из жизни:* Стек — это твой личный блокнот на рабочем столе. Ты быстро написал там номер телефона, позвонил и сразу выкинул листок (мгновенно и бесплатно). Куча — это официальный архив компании. Чтобы положить туда документ, нужно заполнить формы, а потом уборщику (GC) придется проверять, нужен ли этот документ кому-то еще.

Если ты создаешь переменную внутри функции и возвращаешь указатель на неё (`return &user`), компилятор понимает, что переменная "сбегает" (escapes) из функции, и переносит её в кучу.

Ты можешь проверить свой код, запустив компиляцию со специальным флагом:
`go build -gcflags="-m"`

```go
package main

type User struct {
    Name string
}

// Переменная u не покидает функцию. Она останется в быстром Стеке.
func stayOnStack() {
    u := User{Name: "Дмитрий"} 
    _ = u
}

// Мы возвращаем указатель. Переменная u "сбежит" (escape) в Кучу!
func escapeToHeap() *User {
    u := User{Name: "Дмитрий"}
    return &u 
}

func main() {
    stayOnStack()
    escapeToHeap()
}
```
*Совет эксперта:* Старайся избегать необоснованного выделения памяти в куче в "горячих" циклах. Чем меньше мусора летит в кучу, тем реже запускается GC, тем быстрее работает твой сервер.

#### 3. Профилирование с помощью `pprof`
Даже лучший инженер иногда пишет медленный код. Чтобы не гадать, где именно "тормозит" программа, в Go встроен мощнейший рентгеновский аппарат — пакет `pprof`.

Он позволяет в реальном времени посмотреть:
* Какие функции сжигают больше всего процессорного времени (CPU).
* Где именно выделяется больше всего памяти (Heap).
* Какие горутины зависли и ждут ответа (Goroutines).

Чтобы включить `pprof` в своем веб-сервере, достаточно добавить один пустой импорт:
```go
import (
    "net/http"
    _ "net/http/pprof" // Магический импорт, который сам регистрирует роуты для pprof
)

func main() {
    // Поднимаем сервер на порту 8080.
    // pprof будет доступен по адресу http://localhost:8080/debug/pprof/
    http.ListenAndServe(":8080", nil) 
}
```

Как анализировать данные? В терминале ты вызываешь встроенную утилиту:
`go tool pprof -http=:8081 http://localhost:8080/debug/pprof/profile?seconds=30`
Эта команда 30 секунд "слушает" твой работающий сервер, а затем открывает в браузере красивый интерактивный график (Flame Graph), где наглядно видно, какая конкретно строчка кода "съедает" процессор.

#### 4. Метрики и Распределенный трейсинг (Observability)
Когда у тебя один сервер, логи (`fmt.Println`) работают отлично. Когда у тебя микросервисная архитектура из 20 разных сервисов (Auth, Billing, Notifications...), логи становятся бесполезны. 

Запрос от клиента может пройти через 5 разных сервисов. Если на 4-м шаге произошла задержка в 5 секунд, ты никогда не найдешь виноватого, просто читая текстовые файлы.

Для решения этой проблемы Highload-инженеры используют **Observability (Наблюдаемость)**, состоящую из двух столпов:

**А. Метрики (Prometheus):**
Твой код не просто пишет логи, он инкрементирует счетчики. 
Например: `http_requests_total` увеличивается на 1 при каждом запросе. Затем система мониторинга (Grafana) строит графики. Ты сразу видишь, что у тебя выросло количество 500-х ошибок или увеличился перцентиль задержки ($p99$).

**Б. Распределенный трейсинг (OpenTelemetry):**
Когда клиент делает запрос, API Gateway (входная точка) генерирует уникальный `TraceID` (например, `abc-123`). Этот ID передается через тот самый `context.Context` (из Главы 10) во все остальные микросервисы.



Каждый микросервис записывает свой "Span" (отрезок времени) и отправляет его в систему вроде Jaeger или Zipkin. В итоге ты открываешь веб-интерфейс и видишь весь путь запроса:
* `API-Gateway`: 10ms
  * `Auth-Service`: 50ms (проверка токена)
  * `Billing-Service`: 2000ms (ОШИБКА: долгий ответ базы данных)
    * `Database`: 1990ms (запрос `SELECT * FROM invoices`)

Ты мгновенно видишь, что проблема не в коде Go, а в том, что в таблице `invoices` не хватает SQL-индекса!
