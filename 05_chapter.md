### Глава 5. Производные типы и структура памяти

В Go нет классического объектно-ориентированного программирования с классами, конструкторами и иерархией наследования, к которым многие привыкли в языках вроде Java или C#. Go разделяет данные (структуры) и поведение (методы). Это делает код более предсказуемым и избавляет от проблемы "тяжелых" объектов, тянущих за собой ненужный контекст.

#### 1. Пользовательские типы и Псевдонимы
В Go можно создавать новые типы на основе базовых. Это мощный инструмент для защиты от ошибок на этапе компиляции.

*Пример из мира финансов:* У нас есть ID транзакции и сумма транзакции. И то, и другое — числа `int64`. Если передать их в функцию как обычные `int64`, можно случайно перепутать местами аргументы. Пользовательские типы решают эту проблему.

```go
package main

import "fmt"

// Создаем НОВЫЙ тип на основе int64. 
// Теперь TransactionID и Amount - это совершенно разные типы для компилятора.
type TransactionID int64
type Amount int64

// Псевдоним (Alias) - обозначается знаком равно "=". 
// Это просто другое имя для ТОГО ЖЕ САМОГО типа. Используется крайне редко 
// (в основном для рефакторинга больших кодовых баз).
type Currency = string 

func process(id TransactionID, money Amount) {
    fmt.Printf("Обработка транзакции %d на сумму %d\n", id, money)
}

func main() {
    var id TransactionID = 100500
    var money Amount = 5000
    
    process(id, money) // Всё отлично
    
    // process(money, id) // ОШИБКА КОМПИЛЯЦИИ! Нельзя передать Amount вместо TransactionID
}
```

#### 2. Структуры (`struct`) и вложенность
Структура — это способ сгруппировать несколько переменных (полей) разных типов под одним именем. Это прямая альтернатива классам для хранения состояния.

```go
// Определяем структуру клиента
type Client struct {
    ID        int
    Name      string
    IsBlocked bool
}

// Вложенные структуры: структура как поле другой структуры
type BankAccount struct {
    AccountNum string
    Balance    Amount
    Owner      Client // Вкладываем структуру Client внутрь BankAccount
}

func main() {
    // Инициализация структуры
    acc := BankAccount{
        AccountNum: "RU123456789",
        Balance:    150000,
        Owner: Client{
            ID:        1,
            Name:      "Дмитрий",
            IsBlocked: false,
        },
    }

    // Доступ к полям через точку
    fmt.Println(acc.Owner.Name)
}
```

#### 3. Выравнивание структур в памяти (Struct Alignment и Memory Padding)
Это экспертная тема, критически важная для систем, чувствительных к потреблению памяти.

Процессор читает память не по 1 байту, а "машинными словами". На 64-битной архитектуре одно машинное слово = 8 байт. Чтобы процессору было удобно и быстро читать данные, компилятор Go выравнивает поля структуры так, чтобы они ровно ложились в эти 8-байтные блоки. Если поле меньше, компилятор добивает пустоту нулями (padding).



Посмотрим на плохую структуру:
```go
type BadStruct struct {
    IsActive bool    // 1 байт. Занимает начало слова. Остается 7 байт.
    Score    int64   // 8 байт. Не влезает в остаток первого слова! 
                     // Компилятор добавляет 7 ПУСТЫХ байт (padding) к первому слову,
                     // и переносит Score в новое 8-байтное слово.
    IsAdmin  bool    // 1 байт. Начинает третье слово. Плюс 7 пустых байт.
}
// ИТОГО: 1+7 (пустых) + 8 + 1+7 (пустых) = 24 байта!
```

А теперь просто переставим поля по убыванию размера (хорошая структура):
```go
type GoodStruct struct {
    Score    int64   // 8 байт. Занимает ровно первое машинное слово.
    IsActive bool    // 1 байт. Начинает второе слово.
    IsAdmin  bool    // 1 байт. Ложится рядышком со вторым словом.
                     // Остается 6 пустых байт в конце (структура выравнивается по самому большому полю).
}
// ИТОГО: 8 + 1 + 1 + 6 (пустых) = 16 байт!
```
Мы сэкономили 8 байт (треть размера!) просто поменяв строчки местами. Если у нас в кэше сервиса лежат миллионы таких структур, мы экономим мегабайты оперативной памяти и, что еще важнее, снижаем нагрузку на процессорный кэш (L1/L2). Для автоматической проверки структур существует встроенный линтер `fieldalignment`.

#### 4. Методы и Методы указателей
Поскольку в Go нет классов, методы привязываются к структурам (или к любым пользовательским типам) через специальный параметр, который называется **Receiver** (получатель). Он пишется в круглых скобках перед именем функции.

Здесь мы возвращаемся к концепции из Главы 4 (Указатели). Существует два типа ресиверов:

**A. Value Receiver (Получатель по значению)**
Метод работает с *копией* структуры. Любые изменения внутри метода не повлияют на оригинал. Используется, когда метод только "читает" данные (например, вывод на экран или расчеты).

**B. Pointer Receiver (Получатель по указателю)**
Метод работает с *адресом* структуры. Используется в 90% случаев, так как позволяет изменять (мутировать) исходные данные и не тратит процессорное время на копирование всей структуры.

```go
type User struct {
    Name  string
    Email string
}

// 1. Метод с Value Receiver (по значению). Переменная u - это КОПИЯ.
func (u User) GetInfo() string {
    return u.Name + " (" + u.Email + ")"
}

// 2. Метод с Pointer Receiver (по указателю). Переменная u - это УКАЗАТЕЛЬ на оригинал.
func (u *User) UpdateEmail(newEmail string) {
    u.Email = newEmail // Меняем поле у оригинальной структуры
}

func main() {
    user := User{Name: "Alex", Email: "alex@old.com"}

    fmt.Println(user.GetInfo()) // Читаем данные

    // Go автоматически берет адрес переменной user под капотом, 
    // нам не нужно писать (&user).UpdateEmail(...)
    user.UpdateEmail("alex@new.com") 

    fmt.Println("Новый email:", user.Email) // Выведет: alex@new.com
}
```

*Золотое правило инженера на Go:* Не смешивай типы ресиверов. Если хотя бы один метод в структуре требует указателя (для мутации данных), делай ВСЕ методы этой структуры с Pointer Receiver для единообразия интерфейса.


Во второй части **Главы 5** мы заглянем под капот двух самых используемых структур данных в Go: срезов (slices) и хэш-таблиц (maps). Понимание того, как они работают с памятью — обязательный навык для написания высокопроизводительного кода.

#### 5. Срезы (Slices) и их устройство под капотом
Как мы выяснили в Главе 2, массивы в Go имеют жестко фиксированный размер (`[5]int`), что делает их неудобными для большинства реальных задач. Для работы с динамическими списками используются **Срезы (Slices)**.

*Пример из жизни:* Массив — это парковка на 100 мест. Она уже построена, заасфальтирована, её размер не изменить. Срез — это шлагбаум с будкой охранника, который управляет этой парковкой. Сам срез не хранит машины, он только знает, где начинается парковка, сколько мест занято сейчас и сколько всего мест доступно.

Под капотом срез — это крошечная структура (`SliceHeader`), состоящая всего из трех полей (на 64-битной системе она занимает 24 байта):
1. **Pointer (Указатель):** Адрес начала так называемого **Backing Array** (базового массива), где физически лежат данные.
2. **Length (Длина - `len`):** Сколько элементов реально сейчас находится в срезе.
3. **Capacity (Вместимость - `cap`):** Сколько всего элементов может вместить базовый массив, прежде чем придется выделять новую память.



```go
package main

import "fmt"

func main() {
    // 1. Инициализация через make. 
    // Мы создаем срез типа []int (размер в квадратных скобках не указывается!).
    // Длина (len) = 3, Вместимость (cap) = 5.
    nums := make([]int, 3, 5)
    
    // В памяти выделился массив на 5 элементов [0, 0, 0, 0, 0].
    // Но срез смотрит только на первые 3.
    fmt.Printf("len=%d cap=%d %v\n", len(nums), cap(nums), nums) // len=3 cap=5 [0 0 0]

    // 2. Функция append (добавление элементов)
    nums = append(nums, 10) // Добавляем 4-й элемент. Место еще есть (cap=5)
    nums = append(nums, 20) // Добавляем 5-й элемент. Место закончилось!
    
    fmt.Printf("len=%d cap=%d %v\n", len(nums), cap(nums), nums) // len=5 cap=5 [0 0 0 10 20]

    // 3. Что произойдет при следующем append?
    nums = append(nums, 30) 
    // Места больше нет. Go автоматически делает следующее:
    // 1. Выделяет в памяти НОВЫЙ базовый массив большего размера (обычно x2).
    // 2. Копирует старые данные в новый массив.
    // 3. Добавляет новый элемент (30).
    // 4. Возвращает обновленный заголовок среза.
    
    fmt.Printf("После реалокации: len=%d cap=%d\n", len(nums), cap(nums)) // len=6 cap=10
}
```

**Оптимизация для экспертов:** Операция выделения новой памяти и копирования старого массива (реалокация) — очень тяжелая для процессора. Если ты заранее знаешь, что тебе нужно загрузить из базы данных 1000 записей, **всегда** инициализируй срез с заранее заданным `capacity`: `make([]Record, 0, 1000)`. Это позволит избежать десятков дорогостоящих реалокаций в процессе `append`.

#### 6. Карты (Maps) и устройство хэш-таблиц
Срез — это поиск по индексу. Но если нам нужно найти значение по ключу (например, email пользователя), перебирать весь срез долго. Для этого нужны хэш-таблицы, которые в Go называются `map` (карта). Сложность поиска в них в среднем $O(1)$ — то есть мгновенно, независимо от размера карты.

```go
func main() {
    // Инициализация карты, где ключ - string, а значение - int
    // Всегда используй make для инициализации карты! 
    // var m map[string]int создаст nil map, при записи в которую программа упадет (panic).
    balances := make(map[string]int)

    balances["Alex"] = 5000 // Запись
    
    // Чтение. Идиома "Comma, ok" (Запятая, ок).
    // Если ключа "Bob" нет в карте, Go не выбросит ошибку, а вернет нулевое значение (0).
    // Чтобы отличить реальный баланс 0 от отсутствующего ключа, используется вторая переменная (ok).
    val, ok := balances["Bob"]
    if !ok {
        fmt.Println("Пользователь Bob не найден")
    } else {
        fmt.Println("Баланс Bob:", val)
    }

    delete(balances, "Alex") // Удаление по ключу
}
```

**Что под капотом (Архитектура мапы в Go):**
Хэш-таблица в Go — это сложный механизм. 
1. **Хэш-функция:** Когда ты кладешь ключ `"Alex"`, Go пропускает его через математическую хэш-функцию, превращая строку в огромное число.
2. **Бакеты (Buckets):** Память мапы разбита на "ведра" (buckets). Младшие биты хэша определяют, в какое именно ведро попадет наша запись. Одно ведро в Go вмещает ровно 8 пар "ключ-значение".



3. **Коллизии:** Что если два разных ключа попали в одно ведро? Они просто ложатся рядом внутри ведра (до 8 штук). Если ведро переполнено, создается "overflow bucket" (дополнительное ведро), которое привязывается к первому как вагон поезда (связный список).
4. **Эвакуация (Evacuation):** Если мапа понимает, что ведер стало слишком мало, а переполненных вагонов слишком много (Load Factor > 6.5), она удваивает количество ведер. Но она не переносит все данные мгновенно (это повесило бы программу). Go переносит данные "лениво" — по два ведра за каждую новую операцию записи, размазывая нагрузку во времени.

**Важное правило:** Карты в Go **не упорядочены**. Если ты будешь перебирать карту через цикл `for k, v := range m`, порядок выдачи ключей будет абсолютно случайным при каждом запуске программы. Это сделано разработчиками языка намеренно, чтобы программисты не завязывали логику на порядок ключей.
