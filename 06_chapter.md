### Глава 6. Пакеты и модули

#### 1. Пакеты и их импорт (Область видимости)
**Пакет (Package)** в Go — это просто папка на жестком диске, внутри которой лежат файлы с расширением `.go`. Все файлы внутри одной папки принадлежат одному пакету и могут свободно использовать переменные и функции друг друга, даже если они не импортированы явно.

Главная фишка Go — **определение видимости (инкапсуляции) через регистр первой буквы**.
Здесь нет ключевых слов `public` или `private`.
* Если имя функции, переменной или структуры начинается с **Заглавной** буквы (например, `User`, `Calculate()`) — это Экспортируемое (Exported) имя. Оно доступно из других пакетов.
* Если имя начинается со **строчной** буквы (например, `user`, `calculate()`) — это Неэкспортируемое (Unexported) имя. Оно "приватно" и доступно только внутри этого пакета.



*Пример из жизни:* Пакет — это ресторан. Зал для гостей (`PublicMenu()`) доступен всем — это имена с большой буквы. Кухня, подсобка и рецепты шеф-повара (`cookSoup()`) — это внутренняя кухня, скрытая от посторонних глаз (имена с маленькой буквы).

```go
// Файл: mathutils/calc.go
package mathutils // Имя пакета обычно совпадает с именем папки

// Эта переменная приватная, её нельзя прочитать из main.go
var defaultMultiplier = 2 

// Эта функция публичная, её можно вызвать снаружи
func Multiply(a, b int) int {
    return a * b * defaultMultiplier
}
```

```go
// Файл: main.go
package main

// Импортируем наш пакет (путь указывается от корня модуля)
import (
    "fmt"
    "myproject/mathutils" 
)

func main() {
    // Вызываем публичную функцию из другого пакета
    result := mathutils.Multiply(5, 5) 
    fmt.Println(result)
    
    // fmt.Println(mathutils.defaultMultiplier) // ОШИБКА КОМПИЛЯЦИИ! Скрыто.
}
```

#### 2. Разделение проекта на пакеты (Standard Layout)
В Go нет жесткого стандарта структуры папок, навязанного компилятором (кроме папки `internal`, о которой ниже). Но индустрия выработала де-факто стандарт (Standard Go Project Layout), которому следуют почти все серьезные open-source проекты и микросервисы.

* `cmd/` — здесь лежат точки входа (файлы `main.go`). Если у тебя несколько микросервисов в одном репозитории, они будут лежать тут: `cmd/api/main.go`, `cmd/worker/main.go`.
* `pkg/` — публичный код. Библиотеки, которые ты разрешаешь импортировать другим разработчикам в их проектах.
* `internal/` — **приватный код приложения**. Это магическая папка, поведение которой зашито в сам компилятор Go! Код внутри папки `internal` может импортировать только тот проект, в котором она находится. Если кто-то попытается скачать твой репозиторий и импортировать пакет из `internal`, компилятор выдаст ошибку. Это защита твоей бизнес-логики.

#### 3. Модули и Загрузка внешних зависимостей
Пакеты группируются в **Модули**. Модуль — это коллекция пакетов, которая версионируется как единое целое. Корень модуля определяется наличием файла `go.mod`.

**Как начать проект и добавить стороннюю библиотеку:**
1.  **Инициализация:** Выполняешь в терминале `go mod init github.com/твое_имя/твой_проект`. Создается файл `go.mod`.
2.  **Загрузка зависимости:** Тебе понадобился мощный логгер (например, `zap` от Uber). Ты пишешь в терминале: 
    `go get go.uber.org/zap`
3.  **Что происходит под капотом:**
    * Go скачивает библиотеку в глобальный кэш на твоем компьютере.
    * Добавляет строку с версией библиотеки в файл `go.mod`.
    * Создает файл **`go.sum`**. Это криптографический хэш-слепок всех скачанных файлов. 
    *Пример из жизни:* `go.mod` — это список покупок ("Купить молоко 'Домик в деревне'"). `go.sum` — это гарантия того, что по пути из магазина никто не вскрыл пакет с молоком и не подмешал туда яд (гарантия защиты от supply-chain атак, когда хакеры меняют код в популярных Github-репозиториях).

#### 4. Управление зависимостями с помощью папки vendor
В крупных компаниях и финтехе серверы сборки (CI/CD) часто работают в изолированных контурах без доступа к внешнему интернету (или доступ ограничен). Если Github внезапно упадет, проект должен собраться.

Для этого используется команда:
`go mod vendor`

Она берет все сторонние библиотеки из глобального кэша и физически копирует их исходный код в папку `vendor/` внутри твоего проекта. Ты коммитишь эту папку в Git. Теперь твой проект абсолютно автономен: чтобы собрать его на чистом сервере, даже интернет не нужен — компилятор сам увидит папку `vendor` и возьмет код оттуда.

#### 5. Создание разделяемого модуля
Если ты написал классную библиотеку и хочешь, чтобы ей пользовался весь мир, в Go это делается элементарно. Нет никаких централизованных хранилищ вроде `npm` в Node.js или `Maven Central` в Java. **Сам репозиторий (например, Github) и есть хранилище пакетов.**

1.  Создаешь репозиторий на Github.
2.  Пишешь код, убеждаешься, что имена публичных функций начинаются с заглавной буквы.
3.  Делаешь пуш в ветку `main`.
4.  Ставишь Git Tag с семантической версией (например, `v1.0.0`) и пушишь тег.

Всё! Любой человек в мире теперь может написать `go get github.com/твое_имя/твоя_библиотека@v1.0.0`, и компилятор Go сам скачает её напрямую с Github.
