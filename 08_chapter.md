### Глава 8. Интерфейсы и Архитектура

#### 1. Введение в интерфейсы и Duck Typing (Неявная реализация)
Интерфейс в Go — это тип, который описывает **поведение** (набор методов), но не содержит в себе данных или реализации.

Главное архитектурное отличие Go от большинства других языков — **отсутствие ключевого слова `implements`**. Тебе не нужно жестко связывать структуру с интерфейсом. В Go используется концепция **Duck Typing** (Утиная типизация).

*Пример из жизни:* Если что-то выглядит как утка, плавает как утка и крякает как утка — это утка. 
Если у нас есть интерфейс `Кофеварка` с методом `СваритьКофе()`, и мы создаем структуру `УмныйЧайник`, у которой тоже есть метод `СваритьКофе()`, то компилятор Go автоматически посчитает, что `УмныйЧайник` является `Кофеваркой`. 



Это дает невероятную гибкость. Ты можешь создать интерфейс в своем микросервисе, и любая структура из чужой библиотеки автоматически под него подойдет, если у нее совпадут имена и сигнатуры методов, при этом авторы той библиотеки даже не будут знать о существовании твоего интерфейса!

```go
package main

import "fmt"

// 1. Объявляем интерфейс. Он описывает только поведение.
type PaymentMethod interface {
    Pay(amount float64) error
}

// 2. Создаем структуру Банковской Карты
type CreditCard struct {
    CardNumber string
}

// 3. Реализуем метод Pay для CreditCard. 
// Теперь CreditCard АВТОМАТИЧЕСКИ удовлетворяет интерфейсу PaymentMethod.
func (c CreditCard) Pay(amount float64) error {
    fmt.Printf("Оплата %.2f через карту %s\n", amount, c.CardNumber)
    return nil
}

// 4. Создаем структуру Криптокошелька
type CryptoWallet struct {
    Address string
}

// И для него тоже реализуем метод Pay
func (cw CryptoWallet) Pay(amount float64) error {
    fmt.Printf("Перевод %.2f на крипто-адрес %s\n", amount, cw.Address)
    return nil
}

// 5. Функция, которая принимает интерфейс (абстракцию), а не конкретную структуру.
func Checkout(p PaymentMethod, amount float64) {
    // Внутри мы не знаем, карта это или крипта. Мы просто вызываем метод.
    p.Pay(amount) 
}

func main() {
    card := CreditCard{CardNumber: "1234-5678"}
    wallet := CryptoWallet{Address: "0xABCDEF"}

    // В функцию Checkout можно передать любой объект, который "умеет" платить
    Checkout(card, 1500.50)
    Checkout(wallet, 42.00)
}
```

*Золотое правило Go:* **"Принимай интерфейсы, возвращай структуры"**. Твои функции должны требовать минимум (интерфейс из 1-2 методов), а возвращать конкретные типы со всеми их возможностями. Идеальный интерфейс в Go состоит всего из одного метода (например, стандартные `io.Reader` и `io.Writer`).

#### 2. Реализация интерфейсов для указателей структур
Если метод структуры реализован через **Pointer Receiver** (получатель по указателю, например `func (c *CreditCard) Pay()`), то интерфейсу будет удовлетворять **только указатель** на эту структуру (`&card`), но не само значение. 

Компилятор строго за этим следит, потому что метод ожидает адрес памяти для возможного изменения данных, и если передать копию, поведение будет непредсказуемым.

#### 3. Проверка типа интерфейса (Type Assertion и Type Switch)
Иногда функция принимает пустой интерфейс `any` (или `interface{}` в старых версиях Go), который может хранить вообще всё что угодно. Или мы приняли `PaymentMethod`, но нам срочно нужно узнать, а банковская ли это карта, чтобы прочитать её номер.

Для этого используется извлечение типа (Type Assertion). Оно очень похоже на чтение из мапы.

```go
func printCardNumber(p PaymentMethod) {
    // Пытаемся извлечь CreditCard из интерфейса p.
    // Если p действительно содержит CreditCard, ok будет true.
    card, ok := p.(CreditCard) 
    if !ok {
        fmt.Println("Это не банковская карта!")
        return
    }
    // Здесь мы имеем доступ к полям конкретной структуры
    fmt.Println("Номер карты:", card.CardNumber) 
}

// Если нужно проверить сразу несколько вариантов, используется Type Switch:
func analyzePayment(p PaymentMethod) {
    switch v := p.(type) { // Ключевое слово type используется только в switch
    case CreditCard:
        fmt.Println("Обрабатываем карту:", v.CardNumber)
    case CryptoWallet:
        fmt.Println("Обрабатываем крипту:", v.Address)
    default:
        fmt.Println("Неизвестный метод оплаты")
    }
}
```

#### 4. Композиция вместо наследования (Embedding)
В языках с классическим ООП, если нам нужен "Мобильный телефон", мы наследуем его от "Телефона", который наследуется от "Электронного устройства". Это создает жесткую, хрупкую иерархию, где изменения в базовом классе ломают потомков.

В Go нет оператора `extends`. Вместо этого мы собираем сложные структуры из простых деталей, как из конструктора Lego. Это называется **Embedding (Встраивание)**.

```go
package main

import "fmt"

// Базовая деталь 1
type Engine struct {
    Horsepower int
}

func (e Engine) Start() {
    fmt.Println("Двигатель запущен. Мощность:", e.Horsepower)
}

// Базовая деталь 2
type Wheels struct {
    Count int
}

// Собираем Автомобиль с помощью композиции
type Car struct {
    Brand  string
    Engine // Анонимное встраивание (указываем только тип, без имени поля)
    Wheels // Анонимное встраивание
}

func main() {
    myCar := Car{
        Brand: "Tesla",
        Engine: Engine{Horsepower: 1020},
        Wheels: Wheels{Count: 4},
    }

    // МАГИЯ ВСТРАИВАНИЯ:
    // Мы можем вызывать методы и поля встроенных структур НАПРЯМУЮ, 
    // как будто они принадлежат самой машине!
    myCar.Start() // Go под капотом вызывает myCar.Engine.Start()
    fmt.Println("Колес:", myCar.Count) // myCar.Wheels.Count
}
```

Благодаря композиции, `Car` не просто переиспользует код `Engine`. Если бы у нас был интерфейс `Starter` с методом `Start()`, наша структура `Car` автоматически начала бы удовлетворять этому интерфейсу, потому что метод `Start()` "поднялся" на уровень выше. Это изящно решает проблему множественного наследования, не создавая запутанных иерархий.
